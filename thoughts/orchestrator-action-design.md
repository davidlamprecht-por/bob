# Orchestrator Action Design - Discussion Notes

**Date:** 2025-12-30

## Context

Working on implementing `ProcessAction()` function, specifically the `ActionAsync` case. Need to design how async actions work while avoiding recursion and keeping the main action loop flat.

## Core Architecture Decisions

### 1. Workflow-Driven Design ✅

**Decision:** Workflows are the master orchestrators, not the action processor.

- Workflows decide control flow and coordination
- Orchestrator is a "dumb" execution engine that just processes action queue
- All results flow back to workflows for decision making
- Workflows can spawn AI, tools, sub-agents, etc. and decide what to do with results

**Example Pattern:**
```
Workflow spawns: AI action with allowed actions [ToolB, ToolC]
  ↓
AI decides it needs info, spawns: ActionAsync [CallToolB, CallToolC] + ContinuationAction
  ↓
Orchestrator processes async actions in parallel
  ↓
ContinuationAction returns results to Workflow
  ↓
Workflow decides: feed results back to AI, or do something else
```

### 2. Action Source Tracking ✅

**Decision:** Add source fields to Action struct to track which workflow spawned it.

```go
type Action struct {
    SourceWorkflow string  // Which workflow spawned this
    SourceActionID string  // Optional: for fine-grained tracking
    // ... other fields
}
```

**Benefits:**
- Enables sub-workflows naturally
- Easy result routing back to originating workflow
- Enables permission boundaries (can narrow permissions, never widen)
- Traceable execution path for debugging

**Propagation:** When workflow spawns action A which spawns async [B, C], source propagates to all.

### 3. Workflow State Management ✅

**Decision:** Workflows own their state via `Context.WorkflowContext` field.

- Workflows can store whatever they need in WorkflowContext
- Orchestrator just passes context around, doesn't interpret it
- Clean separation of concerns
- Flexible - each workflow defines its own state machine

### 4. Generic Action Input ✅

**Decision:** Use a generic `Input` field rather than type-specific fields.

```go
type Action struct {
    ActionType ActionType
    Input      map[string]interface{}  // Generic payload
    // ... other fields
}
```

**Rationale:**
- Don't yet know all action types we'll need
- Keeps Action struct simple
- Can refactor to controlled interface later if needed
- Flexible for experimentation

## Open Questions - Postponed

### Question 3: Async Action Correlation

**Problem:** When ActionAsync spawns [B, C, D], how does the ContinuationAction know which results are which?

**Options Discussed:**

**Option A: Simple Continuation Payload**
```go
type Action struct {
    ContinuationAction *Action
    AsyncResults       map[string]interface{} // Results collected here
}
```
Orchestrator collects results, passes to continuation. Simple but limited.

**Option B: Workflow-Managed Correlation**
```go
type WorkflowContext struct {
    PendingAsync   map[string]bool
    CollectedData  map[string]interface{}
}
```
Workflows track correlation themselves. More flexible, more complex.

**Option C: Tagged Async Actions**
```go
AsyncActions: []TaggedAction{
    {Tag: "tool_b_result", Action: callToolB},
    {Tag: "tool_c_result", Action: callToolC},
}
// Results map by tag: {"tool_b_result": resB, "tool_c_result": resC}
```

**Option D: Index-Based (simplest)**
```go
AsyncActions: []Action  // Just array
// Results by index: [res0, res1, res2]
```

**Decision:** Postponed - need to think through implications and use cases.

**Considerations:**
- How do workflows inspect results?
- What if some async actions fail?
- Does orchestrator or workflow handle collection?
- Need to support both simple and complex coordination patterns

## Design Principles to Keep

- **Flat action loop** - no recursion, everything goes to main queue
- **Workflow control** - workflows decide flow, orchestrator executes
- **Source tracking** - always know where actions came from
- **Avoid premature optimization** - start simple, refactor when patterns emerge

## Notes

- Currently sequential processing: `for len(queue) > 0 { pop, process, extend }`
