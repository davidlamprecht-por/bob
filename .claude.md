# Claude Code Rules for Bob Project

## Strict Task Boundaries

**CRITICAL RULE: Do NOT fix issues beyond the explicit request.**

When asked to perform a specific task:
- Complete ONLY what was explicitly requested
- Do NOT fix compilation errors unless asked
- Do NOT fix function name mismatches unless asked
- Do NOT fix linting issues unless asked
- Do NOT fix formatting issues unless asked
- Do NOT add "improvements" or "while I'm here" changes

**Examples:**

❌ **WRONG:**
```
User: "Add a getter for AIConversation"
Assistant: *Adds getter AND fixes unrelated compilation error*
```

✅ **CORRECT:**
```
User: "Add a getter for AIConversation"
Assistant: *Adds only the getter, even if compilation fails*
```

If there are issues after completing the requested task, STOP and report them. Let the user decide what to fix next.

## Implementation Philosophy

- **VERY SMALL SETS**: Implement changes in tiny, controlled increments to avoid drift
- **User-directed**: The user drives what gets fixed and when
- **No assumptions**: Don't assume what the user wants fixed next

## Running Bob

**CRITICAL RULE: You are NOT allowed to run Bob yourself.**

- ✅ You ARE allowed to build Bob (`go build`)
- ❌ You are NOT allowed to run Bob (`./bin/bob` or any execution)
- The user will run Bob and provide logs/output as needed
- Never use `run_in_background` or any other method to execute Bob
